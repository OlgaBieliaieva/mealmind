# MealMind

> Інтелектуальна система підтримки прийняття рішень для планування сімейного харчування.

MealMind — програмна система, яка допомагає домогосподарствам формувати спільне меню, враховувати індивідуальні потреби членів родини, розраховувати персоналізовані порції та автоматизувати підготовку списку покупок.

Проєкт розробляється як магістерська кваліфікаційна робота за спеціальністю «Інженерія програмного забезпечення».

## Концепція

MealMind — це не просто каталог рецептів, лічильник калорій або застосунок для ведення дієти.

Основна ідея проєкту полягає у створенні системи підтримки прийняття рішень у повсякденній діяльності людини. Система має зменшити кількість ручних операцій і когнітивне навантаження, пов’язане з організацією сімейного харчування.

На відміну від рішень, орієнтованих на одного користувача, MealMind використовує family-first модель: центральним об’єктом системи є домогосподарство як сукупність взаємопов’язаних профілів із власними потребами, обмеженнями та цілями.

## Проблема

Планування сімейного харчування є багатокритеріальним процесом. Під час формування меню необхідно одночасно враховувати:

- склад родини;
- індивідуальні потреби її членів;
- калорійність і харчову цінність страв;
- алергії та дієтичні обмеження;
- розмір порцій;
- повторюваність страв;
- склад продуктів і рецептів;
- необхідні закупівлі.

У більшості домогосподарств ці рішення приймаються вручну одним членом родини. Це потребує часу, створює значне когнітивне навантаження та ускладнює системне планування.

Більшість наявних програмних продуктів вирішують лише окремі частини цієї проблеми: підраховують калорії, зберігають рецепти або формують персональне меню. MealMind має об’єднати ці операції навколо спільного сімейного плану.

## Мета

Розробити програмну систему підтримки прийняття рішень для планування сімейного харчування, яка автоматизує формування меню, персоналізацію порцій для членів родини та створення списків покупок, зменшуючи кількість ручних операцій і когнітивне навантаження користувача.

## Дослідницький контекст

**Об’єкт дослідження:** процеси планування харчування в домогосподарствах.

**Предмет дослідження:** методи, моделі та програмні засоби автоматизації планування сімейного харчування та підтримки прийняття рішень щодо формування меню, персоналізації порцій і організації закупівель.

**Практична цінність:** скорочення часу формування сімейного меню, автоматизація розрахунків порцій і створення основи для подальшого впровадження рекомендаційних та AI-механізмів.

## Дипломний MVP

### Адміністрування каталогу

- керування продуктами, категоріями, брендами та нутрієнтами;

- контроль життєвого циклу й статусів каталогових сутностей.

### Сімейний профіль

- створення сім’ї;
- керування учасниками сім’ї;
- зберігання індивідуальних параметрів;
- харчові вподобання;
- алергії та дієтичні обмеження;
- персональні цілі харчування.

### Пошук і вибір їжі

- пошук продуктів і рецептів;
- фільтрація за категоріями та характеристиками;
- перегляд складу і харчової цінності;
- збереження вибраних продуктів і рецептів.

### Планування харчування

- формування спільного тижневого меню;
- планування за днями та прийомами їжі;
- призначення страв окремим членам родини;
- персоналізація порцій;
- автоматичне масштабування рецептів;
- агрегований розрахунок харчової цінності;
- перегляд плану за днем, прийомом їжі або учасником сім’ї.

### Список покупок

- формування списку на основі сімейного плану;
- агрегація однакових інгредієнтів;
- урахування масштабованих порцій;
- групування позицій для зручної організації закупівель.

## Архітектура

MealMind проєктується як TypeScript-монорепозиторій із модульним монолітом на стороні API.

```text
┌─────────────────────┐       ┌─────────────────────┐
│     Web Client      │       │      Web Admin      │
│  Next.js / React    │       │  Next.js / React    │
└──────────┬──────────┘       └──────────┬──────────┘
           │                             │
           └──────────────┬──────────────┘
                          │ REST API
                 ┌────────▼────────┐
                 │   Express API   │
                 │ Modular Monolith│
                 └────────┬────────┘
                          │
                     ┌────▼─────┐
                     │  Prisma  │
                     └────┬─────┘
                          │
                ┌─────────▼─────────┐
                │     Supabase      │
                │ PostgreSQL / Auth │
                │      Storage      │
                └───────────────────┘
```

### Серверна частина

Backend організовується за принципами Layered Architecture:

```text
Router
  ↓
Controller
  ↓
Application Service
  ↓
Repository / Query
  ↓
Prisma
  ↓
PostgreSQL
```

Основні архітектурні правила:

- transport layer не містить бізнес-логіки;
- controller не звертається до Prisma безпосередньо;
- application services координують сценарії використання;
- repositories відповідають за зберігання та вибірку даних;
- доменні правила мають єдине джерело істини;
- залежності створюються у composition root;
- модулі взаємодіють через явні контракти.

### Клієнтська частина

Frontend будується за принципами Feature-Oriented Architecture:

```text
app/
features/
shared/
```

- `app` відповідає за маршрути й композицію сторінок;
- `features` містить завершені користувацькі сценарії;
- `shared` містить лише повторно використовувані UI-компоненти, інфраструктуру та утиліти.

## Запланована структура репозиторію

```text
mealmind/
├── apps/
│   ├── api/
│   ├── web-admin/
│   └── web-client/
├── packages/
│   ├── db/
│   └── contracts/
├── .github/
├── package.json
├── turbo.json
└── tsconfig.base.json
```

Пакет `contracts` створюється лише тоді, коли спільне використання транспортних контрактів між API та клієнтами буде практично обґрунтоване. Доменна логіка не переноситься у загальний пакет без визначеного власника.

## Розгортання та моніторинг

Запланована production topology:

```text
mealmind.in.ua             → Vercel web-client
admin.mealmind.in.ua       → Vercel web-admin
api.mealmind.in.ua         → Render API
PostgreSQL/Auth/Storage    → Supabase
Error tracking             → Sentry
```

Мінімальний production baseline включає:

- `/health` і `/ready`;
- структуровані API-логи;
- request/correlation ID;
- Sentry error tracking для трьох застосунків;
- source maps і release metadata;
- uptime check;
- alerts на недоступність API;
- контрольовані міграції;
- backup/restore procedure;
- deployment та incident runbook.

## Приватність і безпека

MealMind може обробляти інформацію про харчові вподобання, алергії, антропометричні параметри й інші чутливі дані.

Система проєктується з урахуванням таких правил:

- збираються лише дані, необхідні для роботи функцій;
- доступ до сімейних даних перевіряється через membership;
- клієнтські ідентифікатори не вважаються доказом identity;
- секрети не зберігаються у Git;
- production, staging і local використовують різні дані та credentials;
- персональні дані не передаються до логів або Sentry;
- передбачаються процедури експорту й видалення даних.

MealMind не є медичним виробом, не встановлює діагнози та не замінює консультацію лікаря або кваліфікованого дієтолога.
