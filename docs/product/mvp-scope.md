# Межі MVP MealMind

## Призначення документа

Цей документ визначає функціональні та нефункціональні межі дипломного MVP MealMind.

Вимоги використовуються як основа для планування реалізації, acceptance scenarios і майбутньої матриці простежуваності `requirement → PR → test`.

## Визначення MVP

MVP MealMind — це завершена наскрізна система, яка дозволяє:

1. адміністратору вести каталог продуктів і рецептів;
2. користувачу створювати сімейний контекст;
3. знаходити продукти та рецепти;
4. формувати тижневий план харчування;
5. персоналізувати порції для учасників сім’ї;
6. отримувати агрегований список покупок.

## Функціональні вимоги

### Управління каталогом

#### CAT-01. Довідники

Адміністратор може переглядати та, де це дозволено моделлю, редагувати категорії, бренди, одиниці вимірювання, нутрієнти, типи прийомів їжі та інші довідники.

#### CAT-02. Продукти

Адміністратор може:

- створювати generic і branded products;
- вказувати категорію, бренд, одиниці та нутрієнти;
- редагувати дозволені поля;
- керувати статусом і архівацією;
- завантажувати та видаляти фотографії.

#### CAT-03. Рецепти

Адміністратор може:

- створювати рецепт;
- додавати інгредієнти та кроки;
- вказувати кількість порцій;
- додавати тип, автора, джерело і медіадані;
- переглядати розраховану харчову цінність;
- публікувати або архівувати рецепт.

#### CAT-04. Розрахунок харчової цінності

Система детерміновано розраховує харчову цінність рецепта на основі кількості інгредієнтів і даних продуктів.

Правила одиниць, округлення і поведінки за відсутності даних мають бути задокументовані.

### Користувачі, сім’ї та доступ

#### FAM-01. Ідентифікація користувача

Система перевіряє зовнішній authentication token і пов’язує підтверджену identity з локальним користувачем.

#### FAM-02. Сімейний контекст

Користувач може створити сім’ю, переглядати доступні сім’ї та вибирати активну сім’ю.

#### FAM-03. Учасники сім’ї

Авторизований користувач може створювати й редагувати профілі учасників у межах дозволеної ролі.

Профіль може представляти користувача з обліковим записом або залежного учасника без власного входу.

#### FAM-04. Параметри учасника

Для учасника можуть зберігатися:

- харчові вподобання;
- алергії;
- дієтичні обмеження;
- параметри, необхідні для розрахунку порцій;
- персональні харчові цілі.

Ці дані не трактуються як медичний діагноз або призначення.

#### FAM-05. Ізоляція даних

Кожна операція із сімейними даними перевіряє membership і роль на сервері.

Клієнтський `userId`, `familyId` або інший ідентифікатор не є самостійним доказом права доступу.

### Пошук і вибір їжі

#### DSC-01. Уніфікований пошук

Користувач може одним запитом шукати опубліковані продукти й рецепти.

#### DSC-02. Фільтрація та навігація

Користувач може фільтрувати результати за типом та підтримуваними характеристиками, переходити між сторінками результатів і відкривати deep links.

#### DSC-03. Деталі продукту

Користувач може переглянути основні дані продукту, харчову цінність і пов’язані рецепти.

#### DSC-04. Деталі рецепта

Користувач може переглянути інгредієнти, кроки, кількість порцій, харчову цінність і медіадані рецепта.

#### DSC-05. Вибране

Користувач може додавати та видаляти продукти й рецепти зі списку вибраного відповідно до визначеної user/family policy.

### Планування харчування

#### PLN-01. Тижневий план

Користувач може переглядати план активної сім’ї для вибраного тижня.

#### PLN-02. Представлення плану

План доступний у представленнях:

- за днем;
- за прийомом їжі;
- за учасником сім’ї.

Усі представлення використовують один канонічний стан плану.

#### PLN-03. Додавання позиції

Користувач може додати продукт або рецепт до визначеної дати, прийому їжі та одного чи кількох учасників.

#### PLN-04. Зміна позиції

Користувач може змінити дозволені поля, кількість, статус або видалити заплановану позицію.

#### PLN-05. Планування для кількох учасників і днів

Користувач може підготувати й підтвердити набір позицій для кількох учасників, днів і прийомів їжі без втрати контексту.

#### PLN-06. Персоналізація порцій

Система підтримує окрему кількість або порцію для кожного учасника сім’ї.

#### PLN-07. Масштабування рецептів

Кількість інгредієнтів рецепта масштабується відповідно до запланованої кількості порцій.

#### PLN-08. Агрегація харчової цінності

Система агрегує доступні показники харчової цінності за днем, прийомом їжі та учасником.

Відсутні дані не повинні непомітно трактуватися як нульові.

### Список покупок

#### SHP-01. Формування списку

Користувач може сформувати список покупок на основі вибраного сімейного плану та періоду.

#### SHP-02. Розгортання рецептів

Система перетворює заплановані рецепти на набір необхідних інгредієнтів з урахуванням масштабованих порцій.

#### SHP-03. Прямі продукти

Продукти, додані до плану безпосередньо, також включаються до списку покупок.

#### SHP-04. Агрегація позицій

Однакові продукти з сумісними одиницями вимірювання агрегуються в одну позицію.

Несумісні або невідомі одиниці не об’єднуються з втратою інформації.

#### SHP-05. Групування

Позиції можуть бути згруповані за категоріями продуктів для зручності закупівлі.

#### SHP-06. Простежуваність

Для розрахованої позиції можна визначити, з яких записів плану та рецептів вона була отримана.

#### SHP-07. Стан позиції

Користувач може позначити позицію як придбану або не придбану, якщо збереження такого стану буде включено до затвердженої моделі MVP.

## Нефункціональні вимоги

### NFR-01. Безпека

- сервер перевіряє authentication і authorization;
- секрети не зберігаються в Git;
- service-role credentials не передаються до браузера;
- критичні dependency, code і secret scans не мають невирішених high/critical проблем.

### NFR-02. Приватність

- чутливі профільні дані не потрапляють до логів або Sentry;
- збираються лише дані, необхідні для функцій MVP;
- має бути описано процедури експорту й видалення даних.

### NFR-03. Надійність даних

- критичні зміни виконуються транзакційно;
- повторні запити не створюють неконтрольованих дублікатів;
- правила округлення, одиниць і агрегації мають єдине джерело істини.

### NFR-04. Якість

Репозиторій має автоматизовані перевірки:

- formatting;
- lint без warnings;
- typecheck;
- unit та integration tests;
- production build;
- security scanning.

### NFR-05. Доступність

Ключові сценарії доступні з клавіатури, мають текстові labels і не покладаються виключно на колір або графічне представлення.

### NFR-06. Адаптивність

Користувацький застосунок підтримує mobile-first сценарій без горизонтального переповнення на основних маршрутах.

### NFR-07. Спостережуваність

API має health/readiness endpoints, структуровані логи та request ID. Неочікувані помилки трьох застосунків реєструються без чутливих payloads.

## Поза межами MVP

До MVP не входять:

- автоматичне призначення дієти;
- AI-рекомендації продуктів або страв;
- автоматична генерація оптимального меню;
- повноцінний щоденник харчування;
- прогнозування майбутніх закупівель;
- облік домашніх запасів;
- бюджетна оптимізація;
- урахування сезонності;
- інтеграція із супермаркетами та службами доставки;
- push-сповіщення;
- оплата або замовлення продуктів;
- нативні мобільні застосунки;
- мікросервісна архітектура.

## Критичні наскрізні сценарії

### Сценарій 1. Каталог

Адміністратор створює продукт і рецепт, перевіряє харчову цінність та публікує рецепт.

### Сценарій 2. Сімейне планування

Користувач створює сім’ю, додає учасників, знаходить рецепт і планує його для кількох учасників.

### Сценарій 3. Список покупок

Користувач відкриває сформований план, генерує список покупок і отримує агреговані інгредієнти з урахуванням персоналізованих порцій.

## Definition of Done MVP

MVP вважається завершеним, якщо:

- усі функціональні вимоги або реалізовані, або явно виключені окремим погодженим рішенням;
- критичні сценарії працюють у staging;
- tenant-isolation tests проходять;
- clean clone і clean database setup відтворюються за документацією;
- немає невирішених high/critical security findings;
- документація відповідає фактичній реалізації;
- сформовано матрицю простежуваності `requirement → PR → test`.
